# CardStudio 멀티 페이지 구조 기획안

## 1. 목표

- 기존 명함 제작기는 그대로 유지한다.
- `CardStudio`를 상위 브랜드로 두고, 기능별로 별도 페이지에 진입하게 만든다.
- 추후 `카드뉴스`, `썸네일`까지 확장 가능한 구조로 정리한다.
- 기존 명함 제작 페이지를 건드리더라도 서비스 중단 없이 단계적으로 전환한다.

## 2. 핵심 방향

현재 구조는 `index.html` 하나에 명함 제작기가 통합된 상태다.

이 상태에서 카드뉴스 기능을 같은 페이지에 억지로 섞으면 다음 문제가 생긴다.

- 명함 로직과 카드뉴스 로직이 충돌함
- 상태 구조가 복잡해짐
- 모바일/PC 편집 흐름이 뒤섞임
- SEO를 기능별로 분리하기 어려움
- 유지보수 비용이 빠르게 커짐

따라서 권장 구조는 아래와 같다.

1. `CardStudio` 메인 허브 페이지
2. `명함 만들기` 전용 페이지
3. `카드뉴스 만들기` 전용 페이지
4. 추후 `썸네일 만들기` 전용 페이지

## 3. 최종 정보 구조

### 서비스 구조

- 메인 브랜드: `CardStudio`
- 한글 표기: `카드스튜디오`
- 브랜드 설명: `간편한 무료 명함 · 썸네일 · 카드뉴스 제작`

### 페이지 구조

- `/`
  - 서비스 소개
  - 기능 선택 허브
  - SEO 랜딩
- `/business-card.html`
  - 기존 명함 제작기
- `/card-news.html`
  - 카드뉴스 제작기
- `/thumbnail.html`
  - 추후 추가

## 5. 메뉴 구조

### 공통 헤더 메뉴

- 명함 만들기
- 카드뉴스 만들기
- 추후 썸네일 만들기

### 허브 페이지 카드 메뉴

  - 추후 오픈 예정 또는 준비중

## 6. 페이지별 역할

## A. 메인 허브 페이지

### 목적

- 브랜드 소개
- 검색 유입 랜딩
- 기능 선택 진입점

### 포함 요소

- CardStudio 헤더
- 브랜드 소개 카피
- 기능 카드 2~3개
- 명함/카드뉴스 차이 설명
- 주요 기능 소개
- FAQ
- SEO 본문

### 중요 포인트

- 앱 편집 UI를 메인에서 직접 노출하지 않음
- 검색엔진은 허브 페이지를 먼저 읽고, 기능 페이지는 별도 색인

## B. 명함 만들기 페이지

### 목적

- 기존 기능 유지
- 현재 사용자를 깨지 않게 이전

### 포함 요소

- 현재 명함 제작기 전체
- 상단에 공통 브랜드 헤더만 추가 가능
- 기존 기능, 위저드, 미리보기, 저장 구조 유지

### 원칙

- 1차에서는 기능 수정 최소화
- 페이지 분리만 우선

## C. 카드뉴스 만들기 페이지

### 목적

- 명함과 독립된 카드 기반 콘텐츠 편집기 제공

### 1차 MVP 기능

- 비율 선택
  - 정사각형
  - 세로형
- 기본 템플릿
- 타이틀 위치 조정
- 타이틀 크기 조정
- 이미지 업로드
- 배경 이미지 설정
- 색상 변경
- 간단 도형
  - 사각형
  - 원
  - 선
- 오버레이
- 레이어 순서 변경
- PNG 저장

## 7. 카드뉴스 전용 상태 구조

명함은 `앞면/뒷면` 구조지만 카드뉴스는 `슬라이드/레이어` 구조가 맞다.

### 초안

```js
project = {
  id,
  type: "cardnews",
  format: "square" | "portrait",
  slides: [
    {
      id,
      background: { type, color, imageUrl },
      layers: [
        { id, type: "text", text, x, y, width, fontSize, color, align, z },
        { id, type: "image", src, x, y, width, height, z },
        { id, type: "shape", shape: "rect" | "circle" | "line", x, y, width, height, fill, stroke, z },
        { id, type: "overlay", color, opacity, z }
      ]
    }
  ]
}
```

## 8. 공용화 전략

처음부터 모든 코드를 공용 모듈로 억지 분리하는 건 비추천이다.

### 1차

- 명함 페이지와 카드뉴스 페이지를 물리적으로 분리
- 카드뉴스는 별도 JS로 시작

### 2차

중복이 확인된 부분만 공용화

- 색상 유틸
- 이미지 업로드 유틸
- export 유틸
- drag/resize 유틸
- storage 유틸

### 이유

- 지금은 속도보다 구조 안정성이 우선
- 명함 코드를 무리하게 리팩터링하다가 현재 기능이 깨질 가능성이 큼

## 11. 전환 방식

기존 사용자가 갑자기 깨지지 않게 아래 순서로 간다.

## Stage 1. 명함 페이지 분리

작업:

- 현재 `index.html` 내용을 `business-card.html`로 복제
- 기존 명함 제작기 자산 경로 확인
- 명함 페이지 단독 동작 확인

완료 기준:

- 기존 명함 기능이 `business-card.html`에서 그대로 동작

## Stage 2. 허브 페이지 신설

작업:

- `index.html`을 CardStudio 허브로 전환
- 명함 만들기 / 카드뉴스 만들기 메뉴 추가
- SEO 소개 본문 유지

완료 기준:

- 루트 페이지에서 기능 선택 가능
- 기존 SEO 메타는 허브 기준으로 유지

## Stage 3. 네비게이션 공통화

작업:

- 허브 / 명함 / 카드뉴스 페이지에 공통 헤더 메뉴 추가
- 브랜드 표기 통일

완료 기준:

- 각 페이지 간 이동 구조 정리

## Stage 4. 카드뉴스 MVP 구축

작업:

- `card-news.html`
- `card-news.js`
- 카드뉴스 전용 CSS 또는 기존 CSS 일부 분리

완료 기준:

- 정사각형/세로형 템플릿 편집 가능
- 타이틀, 이미지, 배경, 도형, 오버레이, 레이어 순서, PNG 저장 가능

## 12. 디자인 원칙

- 허브 페이지는 브랜드 페이지처럼 보여야 함
- 명함 페이지는 현재 편집 효율 유지
- 카드뉴스 페이지는 캔버스 중심 UI
- 한 화면에 명함과 카드뉴스 기능을 섞지 않음

## 13. 리스크

### 리스크 1. 기존 index 기반 명함 진입 링크

- 기존에 `/`를 바로 명함 제작기로 쓰던 사용자는 동선이 바뀜
- 해결:
  - 허브에서 명함 바로가기 강하게 배치

### 리스크 2. 공용 CSS 비대화

- 카드뉴스 전용 스타일을 기존 `style.css`에 계속 누적하면 관리가 힘들어짐
- 해결:
  - Stage 4부터는 카드뉴스 CSS 분리 고려

### 리스크 3. JS 상태 충돌

- 카드뉴스 기능을 기존 `main.js`에 섞으면 유지보수성이 급격히 악화됨
- 해결:
  - 별도 `card-news.js`

## 14. 추천 결론

가장 안전한 구조는 아래다.

1. `index.html` = CardStudio 허브
2. `business-card.html` = 현재 명함 제작기
3. `card-news.html` = 신규 카드뉴스 제작기

이 구조가

- 기존 기능 보존
- 확장성
- SEO
- 유지보수

네 가지를 동시에 만족시킨다.
